# CockroachDB Insert Benchmark

This project contains a Python benchmark script for comparing two insert patterns against CockroachDB using `psycopg3` and `psycopg_pool`.

The benchmark is modeled after a transaction shape where each logical transaction inserts:

- 1 row into `table_a`
- 1 row into `table_b`
- 1 row into `table_c`
- 4 rows into `table_d`

The script runs that pattern for 10,000 iterations by default and can compare:

- Non-pipeline inserts inside a transaction
- True `psycopg3` pipeline mode inside a transaction

## Files

- `aggregate_results.py`: combines JSON result files from multiple regional runners
- `benchmark_crdb_pipeline.py`: main benchmark script
- `requirements.txt`: Python dependency list
- `MEMORY.md`: persistent working notes and decisions

## What The Script Tests

For each iteration, the script performs 4 SQL insert statements:

1. Insert 1 row into `table_a`
2. Insert 1 row into `table_b`
3. Insert 1 row into `table_c`
4. Insert 4 rows into `table_d` using one batched `INSERT ... VALUES (...), (...), (...), (...)`

That means each iteration inserts 7 total rows.

## Test Modes

### 1. Non-Pipeline Transaction

This test uses one transaction per iteration and sends the 4 insert statements in order:

```sql
BEGIN;
INSERT INTO table_a ...;
INSERT INTO table_b ...;
INSERT INTO table_c ...;
INSERT INTO table_d ... VALUES (...), (...), (...), (...);
COMMIT;
```

### 2. Pipeline Transaction

This test uses the same 4 insert statements and the same transaction shape, but sends them through true `psycopg3` pipeline mode:

- `with conn.pipeline():`
- `with conn.transaction():`
- queue all 4 inserts before the pipeline is flushed

This reduces round trips between the client and CockroachDB while keeping the transaction logic comparable to the sequential case.

## Concurrency Model

The benchmark supports multiple concurrent workers.

- Each worker uses one persistent connection from a fixed-size `psycopg_pool` connection pool.
- Workers are distributed across one or more Python worker processes.
- Each process owns its own connection pool.
- Total active connections still match the total worker count.
- By default, `--workers 1`, which means one active benchmark connection.
- If you run `--workers 48`, the benchmark will use 48 active pooled connections during the workload phase.
- `--processes 0` means the script chooses a small automatic process count based on CPU count.

This is the model intended for regional load generation on AWS application nodes.

## Workload Modes

The benchmark supports two execution shapes:

- Iteration mode
  Run a fixed number of logical transactions with `--iterations`.
- Duration mode
  Run continuously for a fixed number of seconds with `--duration-seconds`.

Duration mode is useful when you want a sustained throughput number instead of a finite batch test.

## Table Setup

The script can manage the benchmark tables for you, but schema setup is now explicit.

This is important for multi-region testing, because you usually want:

- one region to run schema setup
- all regions to run workload only

Use these flags:

- `--setup-only`
  Drop and recreate the 4 benchmark tables, then exit.
- `--setup-tables`
  Drop and recreate the 4 benchmark tables before running the selected benchmark mode.

Normal benchmark runs do not perform implicit schema setup.

## Schema

Each table uses a UUID primary key generated by the database with `gen_random_uuid()`.

Each table is created as `LOCALITY REGIONAL BY ROW`, so CockroachDB will home inserted rows based on the gateway region that accepted the insert.

The tables include a mix of types such as:

- `UUID`
- `STRING`
- `DECIMAL`
- `TIMESTAMPTZ`
- `DATE`
- `INT8`
- `FLOAT8`
- `BOOL`
- `JSONB`

There are no foreign keys in this benchmark.

Each table also includes a `benchmark_run_id UUID` column. This lets each runner validate only the rows inserted by its own benchmark run, which is important when multiple regions are writing into the same shared tables.

Each benchmark row also includes variable-length payload text so the inserted row payload is roughly in the `250-300 byte` range. The exact size varies a bit by row and table on purpose so the workload looks more realistic.

The schema has also been widened with many more non-key columns so it more closely resembles a customer table that may carry many secondary indexes.

## Index Profiles

Index creation is intentionally separate from table creation.

Available index-profile files:

- `indexes_5.sql`
  Adds 5 secondary indexes per table
- `indexes_10.sql`
  Adds 10 secondary indexes per table

Recommended workflow:

1. Recreate the tables:
   `python3.11 benchmark_crdb_pipeline.py --setup-only`
2. Apply one index profile:
   `cockroach sql --url "$DB_URI" < indexes_5.sql`
3. Run the benchmark

This lets you compare:

- no extra indexes
- 5 extra indexes per table
- 10 extra indexes per table

## Requirements

- Python 3.11 or compatible newer Python
- Network access to CockroachDB
- A `DB_URI` environment variable

Install dependencies:

```bash
python3.11 -m pip install -r requirements.txt
```

## Connection String

The script reads the CockroachDB connection string from `DB_URI`.

Example shape:

```bash
export DB_URI='postgresql://username:password@host:26257/defaultdb?sslmode=verify-full'
```

Accepted URI forms include:

- `postgresql://...`
- `postgres://...`
- `cockroachdb://...`
- `cockroachdb+psycopg://...`

If needed, the script normalizes non-`postgresql://` forms for `psycopg3`.

## Running The Benchmark

Recreate the tables only:

```bash
python3.11 benchmark_crdb_pipeline.py --setup-only
```

After schema changes such as payload-column updates, run `--setup-only` or `--setup-tables` once before benchmarking so the benchmark tables match the current script.

Apply the 5-index profile:

```bash
cockroach sql --url "$DB_URI" < indexes_5.sql
```

Apply the 10-index profile:

```bash
cockroach sql --url "$DB_URI" < indexes_10.sql
```

Recreate the tables, then run only the non-pipeline benchmark:

```bash
python3.11 benchmark_crdb_pipeline.py --setup-tables --mode non-pipeline
```

Recreate the tables, then run only the pipeline benchmark:

```bash
python3.11 benchmark_crdb_pipeline.py --setup-tables --mode pipeline
```

Run both modes back-to-back, recreating the tables before each mode:

```bash
python3.11 benchmark_crdb_pipeline.py --setup-tables --mode both
```

Run only the non-pipeline benchmark against existing tables:

```bash
python3.11 benchmark_crdb_pipeline.py --mode non-pipeline
```

Run only the pipeline benchmark against existing tables:

```bash
python3.11 benchmark_crdb_pipeline.py --mode pipeline
```

Run with multiple workers:

```bash
python3.11 benchmark_crdb_pipeline.py --mode pipeline --workers 16
```

Run with multiple workers split across multiple processes:

```bash
python3.11 benchmark_crdb_pipeline.py --mode pipeline --workers 48 --processes 4
```

Run in sustained duration mode:

```bash
python3.11 benchmark_crdb_pipeline.py --mode pipeline --workers 48 --processes 4 --duration-seconds 60
```

Run a benchmark and write a regional JSON result file:

```bash
python3.11 benchmark_crdb_pipeline.py --mode pipeline --workers 16 --processes 4 --json-out results-us-east.json
```

If you omit `--region-label`, the script will try to read the current CockroachDB region from the active connection using CockroachDB locality metadata. If that is unavailable, it falls back to the machine hostname.

Run both modes against existing tables without resetting them:

```bash
python3.11 benchmark_crdb_pipeline.py --mode both
```

Aggregate result files from multiple regions:

```bash
python3.11 aggregate_results.py results-us-east.json results-us-west.json results-eu-west.json
```

Aggregate local plus remote peer files over passwordless SSH/SCP:

```bash
python3.11 aggregate_results.py pipeline-us-east.json \
  --remote-host 10.0.1.10 \
  --remote-host 10.0.2.10 \
  --remote-file pipeline.json
```

Override the remote directory if needed:

```bash
python3.11 aggregate_results.py pipeline-us-east.json \
  --remote-host 10.0.1.10 \
  --remote-host 10.0.2.10 \
  --remote-file pipeline.json \
  --remote-dir /some/other/path
```

Useful concurrency-related flags:

- `--workers 16`
  Run 16 concurrent workers and therefore 16 active benchmark connections.
- `--processes 4`
  Split those workers across 4 Python worker processes.
- `--max-retries 5`
  Retry retryable CockroachDB transaction errors up to 5 times per transaction.
- `--progress-seconds 5`
  Print progress updates every 5 seconds during benchmark execution.
- `--duration-seconds 60`
  Run a sustained 60-second benchmark instead of a fixed iteration count.

Useful pipeline-related flags:

- `--pipeline-style transaction`
  Pipeline statements within one transaction at a time.
- `--pipeline-style deep`
  Batch multiple transactions into one pipeline sync to push more work per connection.
- `--pipeline-depth 8`
  Number of transactions to batch per sync in deep pipeline mode.

Useful remote-aggregation flags:

- `--remote-host 10.0.1.10`
  Fetch a remote JSON result file from a peer app node. Repeat for additional nodes.
- `--remote-file pipeline.json`
  The JSON filename to fetch from each remote host.
- `--remote-dir /home/ec2-user/Pipeline-Test`
  The remote directory containing the result file.
- `--ssh-user ec2-user`
  The SSH username used by `scp`.

## Output

For each test, the script prints:

- Timestamped progress messages showing what stage is running
- Payload-generation progress at roughly 10% intervals
- Concurrent benchmark progress at a fixed time interval
- Elapsed time and estimated time remaining at each progress checkpoint
- A per-run `benchmark_run_id`
- Worker count and pool size
- Process count
- Retry count
- Iteration count
- Total rows inserted
- Total runtime in seconds
- Transactions per second
- Rows per second
- Average latency
- p95 latency
- p99 latency
- Final row counts for all 4 tables, filtered to only the current benchmark run

At the end it also prints a simple comparison showing:

- Non-pipeline total runtime
- Pipeline total runtime
- Absolute and percentage improvement

If `--json-out` is provided, the script also writes a JSON file containing:

- region label
- benchmark mode
- benchmark run ID
- iterations completed
- rows inserted
- elapsed seconds
- transactions per second
- average ms per iteration
- rows per second
- worker count
- process count
- pool size
- retry count
- average latency
- p95 latency
- p99 latency
- table counts for the current run
- start and finish timestamps

This JSON output is intended to be aggregated later across regions with `aggregate_results.py`.

## In-Memory Test Data

The script generates all test payloads in memory before benchmarking starts.

That keeps each test working from prebuilt logical transactions and avoids mixing payload generation time into the measured insert timings.

## Notes For Demo Use

- The benchmark is designed so the non-pipeline and pipeline tests use the same transaction shape.
- The main difference is how the statements are delivered to the database.
- For multi-region testing, run `--setup-only` or `--setup-tables` from only one region.
- For regional load tests, run `--mode non-pipeline` or `--mode pipeline` from each region without schema setup.
- Give each region a distinct `--region-label` and `--json-out` path so the results are easy to combine afterward.
- If you omit `--region-label`, the script will try to read the region from CockroachDB automatically.
- The `--workers` value is the active connection count during the benchmark run.
- The benchmark now sets `application_name`, which makes it easier to identify the benchmark sessions in CockroachDB.
- Running `--setup-tables --mode both` is the cleanest apples-to-apples comparison from a single region because each mode starts from empty tables.
- The script now emits progress messages during setup and execution, which makes it easier to narrate during a live demo.
- If you are demoing this live, it can be useful to run `--setup-only` first so the audience sees the benchmark starts from a clean multi-region schema.

## Exit Conditions

The script will fail with an error if:

- `DB_URI` is not set
- `psycopg3` is not installed
- the database connection fails
- pipeline mode is not supported by the client/libpq
- final row counts do not match the expected totals
