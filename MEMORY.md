# CockroachDB Python Test Harness Memory

## Date
- 2026-05-11

## User Goal
- Build a Python test script using `psycopg3` and optionally SQLAlchemy for CockroachDB.
- Test inserts into 4 tables: `table_a`, `table_b`, `table_c`, and `table_d`.
- Per iteration:
- Insert 1 row into `table_a`
- Insert 1 row into `table_b`
- Insert 1 row into `table_c`
- Insert 4 rows into `table_d`
- Primary keys can be UUIDs generated by the database via `gen_random_uuid()`.
- Each table should have 5-6 mixed-type columns.
- Create in-memory test data for 10,000 iterations.
- Benchmark two patterns across 10,000 iterations:
- Test 1: one transaction with sequential inserts
- Test 2: send all inserts at once so CockroachDB can pipeline them
- Keep notes and progress in this file so work can be resumed after a VS Code restart.

## Confirmed Decisions
1. The script should drop and recreate the 4 benchmark tables.
2. No foreign keys are necessary.
3. Connection info comes from the `DB_URI` environment variable.
4. Use true `psycopg3` pipeline mode for the pipelined benchmark.
5. Output should include:
   - total runtime
   - average per iteration
   - rows per second
6. The script will be moved to and run on a Linux box in AWS, not the current Windows desktop.
7. For the next benchmark version, table setup should be invokable explicitly from code with a `setup-tables` style flag before running tests.
8. For the next benchmark version, test selection should be explicit with a flag like `pipeline` vs `non-pipeline` instead of always running both.
9. The user does not intend to run anywhere near 192 workers per region; prefer moderate concurrency targets.
10. In multi-region usage, schema setup should run from only one region; normal test runners in every region should execute workload only.
11. Benchmark tables should be created as `LOCALITY REGIONAL BY ROW` so row placement follows the gateway region of the runner.
12. Because multiple regional runners may write into the same shared tables concurrently, row-count validation must be scoped to the current benchmark run, not the whole table.
13. For cross-region aggregation, each regional runner should write a local JSON results file that can be combined later.
14. Runtime logging should include clear stage messages, progress, elapsed time, and estimated time remaining.
15. Region labeling should default from the CockroachDB connection itself when possible, so the user does not have to pass it manually.
16. The benchmark now needs multi-worker concurrency using a connection pool.
17. The concurrency model should be one persistent pooled connection per active worker.
18. Worker count should be operator-controlled, with a conservative default of 1.
19. The aggregator should optionally fetch peer-region JSON result files over passwordless SSH/SCP.
20. The default remote results directory should be `/home/ec2-user/Pipeline-Test`, but it must be overridable by script parameter.
21. The benchmark should support multiple worker processes, not just one Python process with many threads.
22. The benchmark should support sustained duration mode, not only fixed-iteration mode.
23. Benchmark sessions should set `application_name` so they are easy to identify in CockroachDB session views and metrics.
24. The pipeline benchmark should support an optional deeper batching style to push more work per connection.
25. Benchmark output should include latency stats such as average, p95, and p99.
26. The benchmark rows should now include variable-length payload text so inserted row payload lands roughly in the 250-300 byte range.
27. The benchmark schema should now include many more non-key columns to better resemble a real customer table.
28. Extra secondary indexes should remain separate from table creation and be applied from standalone SQL files.

## Implementation Notes
- The benchmark script uses `psycopg3` plus `psycopg_pool`; SQLAlchemy was not necessary.
- The script accepts `DB_URI` in these forms and normalizes them for psycopg if needed:
  - `postgresql://...`
  - `postgres://...`
  - `cockroachdb://...`
  - `cockroachdb+psycopg://...`
- The script compares:
  - sequential transaction mode
  - true psycopg pipeline mode, with the same 4 insert statements per transaction
- The fourth insert is a single `INSERT ... VALUES (...), (...), (...), (...)` batch into `table_d`.
- The script recreates the tables between the two benchmark runs by default for a fair comparison.
- In-memory payloads are generated once and reused for both tests.
- The script now prints timestamped progress messages during:
  - schema setup
  - payload generation
  - sequential benchmark execution
  - pipeline benchmark execution
  - row-count validation
- The benchmark execution path now supports multiple workers:
  - each worker gets a shard of the prebuilt payloads
  - each worker uses one persistent connection from a fixed-size pool
  - the pool size matches the active worker count
  - progress logging is global across all workers
  - retryable `40001` transaction errors are retried up to a configurable limit
- The benchmark execution path now also supports multiple worker processes:
  - total workers are split across processes
  - each process owns its own connection pool
  - each process labels its sessions with `application_name`
  - progress is aggregated across all processes
- Iteration payloads are now generated on demand instead of prebuilding one giant payload list in memory.
- A sustained duration mode is now supported via `--duration-seconds`.
- Latency stats now include:
  - average latency
  - p95 latency
  - p99 latency
- Pipeline mode now supports:
  - `transaction` style
  - `deep` style with configurable `--pipeline-depth`
- Row payload sizing update:
  - each table now has a `payload_text` column
  - `table_b` also uses a richer `attributes` JSON document
  - payload text lengths vary deterministically by row
  - the goal is roughly `250-300 bytes` of inserted data per row, not an exact physical on-disk size
- Schema widening update:
  - each table now includes many more business-style non-key columns
  - this provides realistic candidate columns for secondary indexing tests
  - separate index profile files were added:
    - `indexes_5.sql`
    - `indexes_10.sql`
  - index counts are now per table:
    - `indexes_5.sql` = 5 secondary indexes on each table
    - `indexes_10.sql` = 10 secondary indexes on each table

## Basic Usage
- Install dependencies:
  - `python -m pip install -r requirements.txt`
- Set connection string:
  - PowerShell: `$env:DB_URI = "postgresql://..."`
- Set connection string on Linux/bash:
  - `export DB_URI='postgresql://...'`
- Recreate only the tables:
  - `python benchmark_crdb_pipeline.py --setup-only`
- Apply 5 indexes:
  - `cockroach sql --url "$DB_URI" < indexes_5.sql`
- Apply 10 indexes:
  - `cockroach sql --url "$DB_URI" < indexes_10.sql`
- Run the full benchmark:
  - `python benchmark_crdb_pipeline.py --iterations 10000`
- Run with multiple workers:
  - `python benchmark_crdb_pipeline.py --mode pipeline --workers 16`
- Run with multiple processes:
  - `python benchmark_crdb_pipeline.py --mode pipeline --workers 48 --processes 4`
- Run in sustained duration mode:
  - `python benchmark_crdb_pipeline.py --mode pipeline --workers 48 --processes 4 --duration-seconds 60`
- Write a per-region JSON result file:
  - `python benchmark_crdb_pipeline.py --mode pipeline --region-label us-east --json-out results-us-east.json`
- Aggregate multiple regional result files:
  - `python aggregate_results.py results-us-east.json results-us-west.json results-eu-west.json`
- Aggregate local plus remote peer files over SSH:
  - `python aggregate_results.py local-results.json --remote-host 10.0.1.10 --remote-host 10.0.2.10 --remote-file pipeline.json`
